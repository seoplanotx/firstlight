# OncoWatch Privacy Modes

OncoWatch is built around a simple privacy boundary:

**Identity stays local. Optional cloud AI receives only minimized, de-identified oncology context.**

This document defines the operating model for Mode 1 and Mode 2.

## Mode 1 — Local-only

Mode 1 is the safest mode and remains fully supported.

In local-only mode:

- Patient/profile details stay on the device.
- Monitoring, matching, scoring, findings, reports, and PDFs run locally.
- OncoWatch can query public sources such as ClinicalTrials.gov and PubMed using profile-derived search terms.
- No LLM/AI provider receives case context.
- Briefings are generated by deterministic local logic.

Use this mode when a user does not want any oncology context sent to an AI provider.

## Mode 2 — De-identified AI assist

Mode 2 is Tucker's intended AI-assisted operating model.

In de-identified AI assist mode:

- Personally identifying details remain local.
- OncoWatch builds a minimized case packet before any cloud AI call.
- Cloud AI may receive cancer type, subtype, stage/context, biomarkers, prior therapy classes, rough region/state, public source text, and matching gaps.
- Cloud AI must not receive names, DOB, exact addresses, phone/email, doctor names, hospital names, local file paths, raw local notes, report PDFs, or persistent patient IDs.

Mode 2 is intended for:

- plain-English summaries of public source material;
- clinician-review questions;
- briefing wording;
- missing-information prompts;
- reducing cognitive load.

Mode 2 is not intended for:

- diagnosis;
- treatment recommendation;
- trial eligibility determination;
- ranking best treatments;
- autonomous clinical decision-making.

## De-identification gate

All cloud AI requests must pass through the backend de-identification gate:

```text
Full local profile / findings
  ↓
backend/app/services/deidentification_service.py
  ↓
De-identified case packet
  ↓
backend/app/services/llm_service.py
  ↓
AI provider
```

No production code should pass raw `PatientProfile`, raw notes, report files, or local storage paths directly into `llm_service`.

## Allowed AI packet categories

Allowed:

- cancer type
- subtype
- stage or clinical context
- current therapy status
- biomarker names/variants/status
- prior therapy names/classes/line/status
- broad location region/state, when needed
- travel radius
- patient preferences/exclusions at a non-identifying level
- public trial/publication title/source/identifier/URL
- public source summaries/snippets
- matching gaps
- safety instructions

Not allowed:

- profile name
- display name
- DOB
- exact address
- phone/email
- doctor or hospital names
- MRN or patient identifiers
- local database/report/file paths
- raw local notes
- full report PDFs
- local persistent profile IDs

## Important caveat

De-identified cancer context can still be sensitive. Rare diagnoses, exact locations, unusual biomarker combinations, and free-text details can increase re-identification risk.

OncoWatch therefore uses three controls:

1. **Minimization** — send only what the AI task needs.
2. **Generalization** — reduce exact location/date details.
3. **Disclosure** — require user acknowledgement before Mode 2 is saved.

## HIPAA wording

Do not claim Mode 2 is automatically HIPAA-compliant.

Acceptable wording:

> OncoWatch keeps identity local by default and can optionally send minimized, de-identified oncology context to an AI provider for summaries and briefing support.

Avoid:

> HIPAA-compliant AI medical processing.

A hosted clinical edition would require additional legal/security controls, likely including BAAs, audit logs, retention controls, access controls, incident response, and a formal risk assessment.
